# Claude Code编码实战

真正决定 AI 编码效果的，不是会不会让它写代码，而是有没有把规则说清楚、把审查做扎实、把工具用成流程。

这次实战围绕一个 `Spring MVC + Maven` 的 TodoList 项目展开，核心结论只有一句话：**AI 负责执行，人负责审核。**

## 一、Claude Code 解决的不是“会不会写”，而是“能不能进项目里写”

和普通 AI 对话相比，Claude Code 的价值在于它能直接进入项目上下文。

- 普通 AI 往往依赖你手动粘贴代码
- Claude Code 可以直接读取项目文件
- 普通 AI 更像给建议
- Claude Code 更像在项目里协助实现、修改和整理代码

它的典型能力包括：

- 根据描述生成代码
- 跨文件修改和重构
- 理解项目结构
- 辅助生成 commit message 和 PR 描述

## 二、先让它搭骨架，再让人把第一道关

在项目初始阶段，可以先让 Claude Code 帮忙完成基础搭建，例如：

- 创建 Maven 项目
- 搭好 `controller / service / model` 目录结构
- 补齐基础配置文件

但项目初始化完成后，必须先审一遍：

- 依赖版本对不对
- 目录结构是否符合团队规范
- 配置是否齐全
- 技术栈是否和现有项目一致

AI 适合先把骨架搭出来，人再负责确认这个骨架是不是能真正落地。

## 三、CLAUDE.md 是项目级规则，不是普通说明文档

`CLAUDE.md` 的作用，是把项目规范明确告诉 AI，让它后续生成代码时尽量按统一规则执行。

一个实用的 `CLAUDE.md`，通常会写清楚这些内容：

- 项目简介和技术栈
- 目录结构约定
- 编码规范
- 接口规范
- 审核要点

以 TodoList 项目为例，可以把规则明确到：

- 所有接口统一使用 `Result<T>` 返回
- DTO 使用 `JSR-303` 参数校验
- 写操作的 Service 方法加 `@Transactional`
- 删除默认使用逻辑删除
- 分页参数有默认值和上限
- 当前用户只能操作自己的数据

`CLAUDE.md` 的价值，不是让 AI 变聪明，而是让 AI 的输出更稳定、更接近项目规范。

## 四、Claude Code 生成代码后，审核要比生成更重要

就算有了 `CLAUDE.md`，也不能把生成结果直接当成最终结果。

在 TodoList 的代码生成过程中，至少要检查：

- 是否真的按统一响应格式返回
- DTO 是否补上了参数校验
- 删除是否走逻辑删除
- 写操作是否加了事务
- 分页参数是否有限制

更关键的是业务层面的判断：

- 有没有做权限校验
- 有没有限制只能操作自己的数据
- 状态流转是否合理
- 是否漏掉边界条件

`CLAUDE.md` 能减少偏差，但不能替代人工审核。

## 五、描述越清晰，Claude Code 产出越稳定

Claude Code 可以理解项目上下文，但业务逻辑仍然依赖输入质量。

如果只是说：

- 帮我写一个创建待办接口

它生成的内容大概率只能做到“能跑”。  
如果把规则说清楚，例如：

- 实现 `POST /api/todos`
- `title` 必填，长度不超过 200
- `categoryId` 可选
- `status=0`
- `userId` 从 `UserContext` 获取
- `is_deleted=0`
- 如果传了分类，要校验分类是否存在

生成结果就会明显更稳定。

所以 AI 编码里有一个非常直接的规律：

- 描述越具体，返工越少
- 描述越模糊，审查成本越高

## 六、Skill 的作用，是把“怎么审”固化下来

如果说 `CLAUDE.md` 是“项目规范”，那 Skill 更像“流程模板”。

它解决的不是“该做什么”，而是“按什么步骤做”。

Skill 最适合拿来固化审查流程。例如把 TodoList 的代码审查 Checklist 固化成一个 Skill，里面可以直接约束：

- 接口层检查统一响应和参数校验
- 业务层检查事务和重复状态判断
- 数据层检查 SQL 风险、逻辑删除和分页限制
- 模型层检查 DTO、VO 和实体字段是否完整

这样做的好处很明显：

- 每次审查角度更稳定
- 每次输出结构更一致
- 审查结果更容易复核

AI 可以按 Skill 做初筛，但真正拍板的仍然是人。

## 七、MCP 的作用，是让 AI 不只看代码，还能读文档、连工具

MCP 可以理解成 AI 连接外部工具和数据源的通道。

接入 MCP 后，Claude Code 不只处理当前代码仓库，还可以进一步连接：

- 文件系统
- Git
- 语雀等文档系统
- 搜索引擎

这会带来很直接的提升：

- 可以直接读取系统文档，不用手动复制粘贴
- 可以结合接口文档和数据模型生成代码
- 可以查询外部规范和参考资料

但 MCP 也会扩大风险范围，常见问题包括：

- 读到不该读的内容
- 执行高风险操作
- 使用了过期文档
- 访问超出权限范围的资源

所以 MCP 使用时，必须守住几条底线：

- 最小权限
- 可审计
- 敏感信息脱敏
- 关键结果交叉验证

## 八、一套更实用的 Claude Code 开发流程

把 Claude Code、`CLAUDE.md`、Skill 和 MCP 串起来后，一套更实用的流程可以是：

1. 先准备 `CLAUDE.md`，把项目规则写清楚
2. 如果有设计文档，再通过 MCP 接入
3. 让 Claude Code 根据规则生成代码
4. 用 Skill 做第一轮审查
5. 人工对照关键项做第二轮审核
6. 有问题就让 Claude Code 定点修正
7. 最后再生成 commit message 和 PR 描述，由人确认后提交

这套流程的关键不在于“AI 一次写对”，而在于：

- 先生成
- 再审查
- 再修正
- 最后提交

## 九、AI 时代，开发者更像审核者

开发者的角色正在变化。

- 以前更多是执行者
- 现在更像审核者和把关者

不管工具怎么变，有三件事始终不会过时：

1. 把需求和规则说清楚
2. 及时审核生成结果
3. 守住安全、业务和质量底线

也可以把这套工具链简单理解成：

- Claude Code：编码助手
- `CLAUDE.md`：项目规范
- Skill：流程模板
- MCP：能力扩展

## 总结

Claude Code 真正有价值的地方，不只是“帮你写代码”，而是它可以和项目规范、审查流程、外部文档一起组成一套更完整的 AI 开发方式。

更合适的使用姿势，不是把判断权交给 AI，而是让 AI 负责执行，让人负责定义规则、审核结果、守住底线。这样 AI 才能真正成为效率放大器，而不是新的风险来源。
